# コーポレートサイト見積・提案キット（Corporate Site Kit）

**HP（コーポレートサイト）** の見積・提案書を、お客様のインプット資料と現行サイトから作るための再利用キット。＝会社の多セクション情報サイト（会社情報・サービス・事例・ニュース・採用など）。

**前提：AIが作業し、人が指示・レビューする。** 各フェーズはAIが起案 → AIがエキスパート役のサブエージェントで自己レビュー → **人間レビューでゲート**。人が目的を決めてFBし、AIが成果物を作る。成果物の出力言語は指示に従う（既定は日本語）。

## このキットが対象とするもの・しないもの

対象＝**HP（コーポレートサイト）**：多セクションの情報サイト。たいていCMS運用で、旧サイトからのコンテンツ統合・移行を伴うことが多い。

- **典型セクション**：会社情報／サービス・製品／事例／ニュース／採用／問い合わせ（＋IR・ブログ・英語ページなど）。
- **典型作業**：セクションをまたぐ情報設計(IA)／旧サイトからのコンテンツ移行（全URL台帳・転送(301)）／少数の共通テンプレへの大量ページ流し込み／一覧の検索・絞り込み／引き渡すCMS。
- **売る価値**：探しやすい・ブランド印象が整う・コンテンツが資産として残る・採用やSEOに効く。

対象外：

- **LP（ランディングページ）**：単一ページの訴求サイト。ページ数が少なくコンバージョン重視・移行/IAの深さがない。フェーズ1〜3の大半が過剰。
- **Webアプリ**：認証・DB・業務ロジック・独自のインタラクティブ機能。見積のドライバはページや移行でなく**実装の複雑さ**。ソフトウェア見積の手法を使うこと。

目安：**ページ・セクション・コンテンツ・移行**が主役→このキットが合う。**機能・ロジック・状態**が主役→合わない。

## 中身

| ファイル | 何か | いつ使う |
|---|---|---|
| `README.md` | これ＝入口（人もAIもまず読む）・全体像（対象・文書の関係・進め方） | 最初に読む |
| `workflow.md` | フェーズ1〜4の手順・共通ルール・落とし穴をステップに埋め込んだ唯一の作業手順書 | READMEの次に開き、上から順に進める |
| `templates/` | 記入する空の骨格 | 作業フォルダにコピーして使う |

## 文書の関係（依存の流れ）

先頭番号＝読む順・作る順。前の文書が次の文書の元になる。**md＝内部用／html＝外部の確認用／PDF＝お客様への提出物（納品）**。

```
01_requirements   →  02_proposal-design  →  03_work-breakdown  →  04_proposal.html  →  04_proposal.pdf
（要件整理・内部）     （提案整理・内部）        （作業リスト・内部）      （外部・確認用）      （お客様提出）
 件数の正              見積モデル・前提の正       金額の正                              └─ 03の外部列だけを
                                                                                          巻末「見積明細」に載せる
```

- **01 → 02**：整理した事実・要件が、提案ストーリー(p)の元になる。
- **02 → 03**：各pの実現方法が、見積を積む作業(w)の元になる。
- **03 → 04**：金額（03が正）を提案書に載せる。内部列（単価・人日・%）は出さず、**外部列だけを巻末「見積明細」に転記**する。
- **母艦は01**：全文書は01の番号で1本につながる（i→r→p/w）。網羅は01の列を数えて確かめる＝詳細は `workflow.md` の「たどれるようにする」。
- **件数は台帳（01/inventory）が唯一の正**：02・03・04・`design/` は台帳の値を**引用するだけ**で数え直さない。再カウントすると同じ件数が文書ごとにズレる（起点の取り方が変わるため）。同一の件数・金額・期間が文書間で一致するかを各★前に突き合わせる。

## 作業フォルダの構成

`templates/` を作業フォルダにコピーして記入する。典型的なレイアウト：

```
<案件>/
├── input/          ← お客様インプット（正・編集禁止）
├── work/           ← ここで成果物を作る
│   ├── 01_requirements.md      … 事実と要件（母艦・件数の正）
│   ├── 02_proposal-design.md   … 提案ストーリー・実現方法・見積モデル・前提
│   ├── 03_work-breakdown.md    … 見積の積み上げ（金額の正）
│   ├── 04_proposal.html        … お客様向け（ここからPDFを書き出す）
│   ├── inventory/              … 現行サイトの全URL台帳（サイトごと1ファイル）
│   ├── design/                 … 02を見積に落とすための調査・展開資料
│   └── img/                    … インプットのレンダー画像（編集禁止・裏どり用）
└── output/<日付>/  ← 提出用に書き出したPDF
```

## 始め方（人がやること）

1. このREADMEを読み、全体像（対象・文書の関係）をつかむ。
2. AIに「このREADMEと `workflow.md` に従って作って」と伝える。
3. AIから質問が来たら答え、各フェーズの**★ゲート**でレビュー・FBする（OKを出すと次へ進む）。

実際の手順（フェーズ1〜4）とAIが守るルールは `workflow.md` 側にある。
