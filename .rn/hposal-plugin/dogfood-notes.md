# hposal dogfood — 改善メモ

`/hposal:up` を実案件（小規模・個人事業のコーポレートサイト、現行 Wix 10ページ）で頭から実走して見つけた、
プラグイン本体（SKILL.md / テンプレ / README）への改善候補。**client PII はここに書かない**（このリポジトリは公開マーケット）。
観察は generic な形で記録する。各項目は〔phase〕〔深刻度 高/中/低〕〔種別 SKILL/template/README〕付き。

> 反映するときは hposal の CHANGELOG `[Unreleased]` と steering（`.rn/hposal-plugin/steering.md`）にも連動させる。
> dogfood の元データ・成果物は公開外（ユーザー private フォルダ）にあり、ここには結論だけ持ち帰る。

## 良かった点（効いた仕組み）

- **機械カウントの徹底**〔phase1〕：sitemap index → 子sitemap を `<loc>` で集計、という SKILL の指示通りにやると母数が一発で固まった。WebFetch で数えない、の戒めが実際に効く場面だった。
- **「目視でなく実査読」**〔phase1〕：連番タイトルのブログ記事を「重複の疑い」で止めず、★を待たず実ページを読んだら**連載シリーズ**と判明。SKILL の「crawl, don't eyeball」「open question を★前に潰す」が母数の振れを縮めた。
- **自己レビュー subagent**〔phase1〕：専門家ロールの subagent が、移行案件特有の穴（301/SEO・toC法務）を実際に拾った。フェーズのループ（draft → expert self-review → ★）が機能している。

## 改善候補

### 1. ★前の標準確認に「301/SEO移行維持」が無い〔phase1〕〔高〕〔SKILL〕

- **症状**: Phase 1 の「Standard topics to confirm before ★」は ①移行線引き ②母数 ③統合の吸収/廃止 ④ニュース移行起点 ⑤依頼の読み の5点。だが**旧URL→新URLの301リダイレクト計画・指名検索/被リンクの引き継ぎ**が無い。今回は subagent レビュー任せで拾えたが、これは移行案件の第一の落とし穴で、施主の課題が「検索で出てくる」だと直撃する。
- **提案**: Phase 1 の標準確認に「⑥ 移行に伴う SEO 資産の維持（301マッピング・サーチコンソール移転）をスコープに含めるか」を追加。あるいは Phase 1 の要件化で「移行案件なら 301/SEO 維持を要件として必ず立てる」を明記。
- **注意**: Phase 2 の migration checklist には「who builds the 301 old→new URL map」がある（実装側）。だが Phase 1 で**要件として顕在化**させる導線が弱い。要件に無いと 02 で拾い損ねる。

### 2. サブドメイン型サービス（Wix/Studio/Ameba 等）の301制約への注意が無い〔phase1-2〕〔中〕〔SKILL〕

- **症状**: `*.wixsite.com` のようなサブドメイン運用は、施主がドメイン権限を持たず**301リダイレクト自体に制約**がある（検索評価が引き継げないリスク）。SKILL の crawl fallback は「JS rendered nav」までは触れるが、この**移行元プラットフォーム固有の301制約**は無い。
- **提案**: Phase 2 migration checklist か Phase 1 の ⚠️ に「移行元がサブドメイン型（独自ドメインでない）の場合、301の可否とSEO引き継ぎ制約を確認」を1行追加。

### 3. toC 化（BtoB→BtoC 転換）案件で要る必須ページのチェックが無い〔phase1-2〕〔中〕〔SKILL/template〕

- **症状**: 個人客向け（予約・将来決済）に踏み出す案件では、**特商法表記・プライバシーポリシー・キャンセルポリシー・有資格表記**が事実上必須。今回 subagent が拾ったが、SKILL/テンプレに「toC要素があるなら法務ページ群を要件に」という観点が無い。コーポレートサイトでも BtoC 寄りだと落ちる。
- **提案**: Phase 2 の「always include」リスト（IA/テンプレ/移行/301/CMS…）に「toC（個人向け取引・予約・決済）があるなら法務ページ（特商法・プライバシー・キャンセル）と資格/許認可表記」を条件付きで追加。

### 4. 「基盤選定そのものが提案物」のケースの足場が薄い〔phase2〕〔中〕〔SKILL/template〕

- **症状**: 今回は CMS が既定でなく「WordPress か Studio か」を**提案で比較して施主が選ぶ**のが肝。SKILL は「the CMS handed over」を前提に進む書きぶりで、**基盤を複数案で比較提示する**型（比較軸：初期費/ランニング/更新運用主体/拡張性/保守の手離れ）の足場が無い。
- **提案**: Phase 2 に「基盤/CMS が未定なら、2–3案を同じ軸（初期・ランニング・更新主体・拡張性・保守）で比較する表を出し、★で選ぶ」パターンを1ブロック追加。テンプレ 02 に比較表の枠があると尚良い。

### 6. セッション継続（ステアリング）が hposal 単体では無く、rn 連携の案内も無い〔全体〕〔中〕〔README〕

- **症状**: hposal の再開は「出力ファイルの有無＋★ゲート」設計で、Goal/Tasks/Decisions/State を持つ**ステアリングは出さない**。これ自体は設計として妥当。だが利用者が「中断・再開・意思決定の記録」を求めると、それは rn の役割だと**どこにも書いていない**（README に rn/steering/セッション/再開 の語ゼロ＝grep 確認）。
- **実例**: dogfood 中に「次回からステアリングは出る？rn のルールは使う？」という当然の疑問が出た。答えは「hposal 単体では出ない。継続したいなら `/rn:gm` で包み、その中で `/hposal:up` を回す」。
- **提案**: hposal README に1〜2行「中断・再開やセッションの意思決定記録が要るなら rn と併用（rn で開始し中で up を回す）」を追記。マーケットに rn と並ぶ前提なので、相互の役割分担（rn=セッション管理／hposal=ドメイン手順）を一言示すと初回の迷いが消える。
- **注意**: これは英語の README（root 一覧行/メタ）でなく hposal/README.md（日本語・利用者向け）に書く話。文言は日本語。

### 7. 「テンプレ基盤だから安い」の罠が Phase 2 見積モデルで防げていない〔phase2〕〔高〕〔SKILL〕

- **症状**: テンプレ型ノーコード基盤（Studio/Wix/ペライチ等）を選ぶと「テンプレ流用＝全部安い」と独り歩きしやすい。実際に圧縮できるのは**初期デザインの起こし方だけ**で、(a)エディタでの構築・レスポンシブ調整 (b)全ページの流し込み (c)予約導線・法務ページの作り込み (d)301/DNS/SC移転 は基盤を問わず工数が残る。専門家レビューで「受注後に最も損が出る穴」と指摘された。SKILL Phase 2 の見積モデル ⚠️群（二層外注・二重課税・年額別建て）にこの「テンプレ＝安いの罠」が無い。
- **提案**: Phase 2 見積モデルの ⚠️ に「ノーコード/テンプレ基盤でも圧縮できるのは初期デザインのみ。構築・流し込み・付随/法務ページ・移行/301 は基盤を問わず別計上」を1行追加。
- **関連**: 同レビューで「取引条件（着手金・検収・軽微修正期間の上限・権利）を 04 前に必ず埋める」「予約と決済を分離（予約=初期/決済=将来）」も出た。前者はテンプレに枠はあるが空のまま進みやすい＝Phase 4 前ゲートに「取引条件の空欄チェック」を足す案。後者は案件依存で汎用化しない。

### 8. 契約形態を AI が勝手に確定してしまった（請負と置いた→施主が準委任に訂正）〔phase2〕〔高〕〔SKILL〕

- **症状**: フェーズ2 見積モデルで AI が「契約形態＝請負（スコープ固定）」と**理由付きで断定**した（「小規模・個人客で何にいくら見える方が安心」）。だが施主の実態は準委任（デザイン・仕様を fix せず、構築しながら変更が出る）。施主指摘で訂正。契約形態は金額の見せ方（総額のみ vs 分類別）まで左右する重大分岐なのに、**q/★ に上げず AI が決めた** ＝ SKILL の大原則「Don't make the human's call」違反。
- **根因**: SKILL は「Match amount granularity to the contract form（準委任→総額のみ／請負→分類別）」と**結果の対応**は書くが、「**契約形態そのものは人間が★で決める q**」と明示していない。AI が見積モデルを「決める」場所なので、つられて契約形態まで決めてしまう。
- **提案**: Phase 2 見積モデルに「⚠️ 契約形態（準委任/請負）は金額の見せ方を左右する＝**AIが決めず q として★へ**。デフォルトで請負を置かない（仕様未確定の Web 制作は準委任が一般的）」を1行追加。標準確認項目に「契約形態」を入れてもよい。
- **効き**: これは「受注後トラブル」でなく「提案の建て付けそのもの」を誤る穴なので深刻度高。実際に dogfood で一発で踏んだ。

### 9. フェーズ3に「役割別集計 ⇄ 明細列 の金額突合」機械チェックが無い〔phase3〕〔高〕〔SKILL〕

- **症状**: フェーズ3実走で、明細の各行を役割別（デザイナー/コーダー）に集計し直す際、**1行（付随ページ・コーダー1人日）をロールアップから落とし**、明細列の合計（¥880k）と役割別集計（¥840k）が¥40k 食い違ったままドラフトを出した。専門家subagentレビューが拾ったが、これは「金額の正＝03」を汚す類の事故で、下流（ディレクション・税・総額・予算突合の見出し額）が全部ズレる。提示する総額そのものが過小になる＝施主との金額会話を誤った数字で始める。
- **根因**: SKILL のトレース節は「各★前に同じ指標（件数・金額・期間）を文書間で突合せよ」と書くが、**03 内部の二つの集計（明細列の総和 と 役割別集計）が一致するか**という最小の機械チェックは無い。フェーズ4には `{{ }}`/`例` 残留ゼロの grep ゲートがあるのに、フェーズ3の金額にはゲートが無い。
- **提案**: フェーズ3完了条件に「**明細列の作業見積合計 ＝ 役割別集計小計 を機械的に突合**（`awk`/電卓で再加算し一致を確認）」を1行追加。人手の集計は1行落ちる前提でゲート化する（フェーズ4の grep ゲートと同じ思想）。
- **効き**: dogfood で一発で踏み、深刻度高。専門家レビュー任せにせず自前で止められる種類。

### 5. 小規模・個人客案件は大規模向け ⚠️ 機械が重い＝「規模に応じた省略可否」の明示が要る〔phase3〕〔中〕〔SKILL〕

- **phase 3 実走の所感（当初の仮を確定）**: 個人事業（10ページ Wix→Studio）でも、移行/301/SC・両面検算・責任分担表は**効いた**（核がコーダーに寄ることが検算で見え、薄い行も炙り出せた）。一方で積み上げは**税込 約¥116万**になり、「コスト抑えたい・明示予算なし」の個人施主には上振れし得る。SKILL は積み上げの網羅は説くが、**規模に応じて省略/段階化してよい候補**（手動テスト範囲・見守り期間・付随ページ・WP比較の深さ）を★で握れ、という導線が弱い。
- **提案**: フェーズ3 の ⚠️ か完了条件に「小規模案件では、積み上げ後に**省略/段階化できる行を施主と握る**（手動テスト範囲・見守り期間・付随ページ等）。レンジ提示なら下限=省略後・上限=フル積み上げ」を1行追加。二層下請け前提の重い検算は外注がある時のみ適用、と条件を添える。

## 反映方針

- 1〜3 は移行/転換案件の頻出落とし穴で深刻度が高い。dogfood 完走後、まとめて SKILL.md へ最小追記（構造変更なし）＋CHANGELOG `[Unreleased]`。
- 4 はテンプレ追加を伴うので別途検討。
- 5 は phase 3 実走で確定済（上記）。SKILL に「規模に応じた省略可否」を1行。
- 8・9 は深刻度高。9（フェーズ3の金額突合ゲート）は phase 4 完走後に 1〜3 とまとめて SKILL.md へ反映。
- **進捗（dogfood）**: phase 1–2 完走、phase 3 ドラフト+専門家レビュー+修正まで完了（★保留）。phase 4 未着手。残り＝phase 3 の★（提示額の丸め/レンジ）→ phase 4（提案書HTML→PDF）。
